# Phase 12 — Periodic expenses

## What was built

- **Subcategory ORM**: Added `is_periodic` (bool, default False) and `due_day` (int, nullable). Migration autogenerated with `uv run alembic revision --autogenerate`; `server_default=sa.false()` added for `is_periodic` so existing rows get a value.
- **Schemas**: SubcategoryRead, SubcategoryCreate, SubcategoryUpdate include `is_periodic` and `due_day`. Create validator: `due_day` required when `is_periodic=true`; `due_day` must be 1–31 when present. Update validator: `due_day` in valid range when provided.
- **Type consistency**: On create and update, `subcategory.belongs_to_income` must match `category.is_income`. Service raises 422 when mismatch.
- **Due-status rule (§3.3)**: Implemented `get_due_periodic_expenses(db, user_id, year, month)` in `app/services/dashboard.py`. Returns list of user’s periodic subcategories with a `paid` flag: **paid** ⇔ at least one transaction for that subcategory in that month. Schema `DashboardDuePeriodicExpenseRead` (subcategory_id, subcategory_name, category_id, category_name, due_day, paid). Phase 13 will add the HTTP endpoint.
- **Tests**: Unit tests for is_periodic/due_day create success and validation error; type consistency 422; due-status (empty, unpaid when no tx in month, paid when tx in month, other month ignored).

## Decisions made

- **due_day ORM typing**: Used plain `mapped_column(Integer, nullable=True)` without `Mapped[int | None]` to avoid SQLAlchemy + Python 3.14 union typing issue (consistent with STATE.md note on optional columns).
- **Migration**: Autogenerate only; edited generated migration only to add `server_default=sa.false()` for `is_periodic` so upgrade succeeds on DBs with existing rows.
- **Due-status in Phase 12**: Business logic and schema implemented; HTTP endpoint GET `/dashboard/due-periodic-expenses` deferred to Phase 13.
